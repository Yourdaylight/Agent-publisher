# 需求文档

## 引言
当前文章发布链路存在两类核心问题：一是正文图片虽然可能已上传到微信侧，但在系统界面的素材库中不可见，导致用户无法确认素材状态；二是文章发布目标被错误地绑定在单一公众号关系上，导致测试与发布必须通过修改 `agent.account_id` 之类的临时手段完成，这不符合业务逻辑。

本需求聚焦于三个目标：
- 让正文相关媒体素材在系统素材库中可见、可追踪；
- 让本地 skill 能够针对远端部署环境完成标准化验证，而不是依赖手工改库或临时绑定；
- 让单篇文章支持面向多个公众号执行独立的发送/草稿同步流程，不再依赖“一个 Agent 只绑定一个公众号”的限制。

## 需求

### 需求 1

**用户故事：** 作为一名内容运营人员，我希望文章正文图片及相关媒体素材能够在系统素材库中被看到，以便我确认素材是否已成功处理并可用于微信发送。

#### 验收标准
1. WHEN 系统为文章正文处理图片素材 THEN 系统 SHALL 为每个素材生成可追踪的素材记录。
2. WHEN 用户打开素材库界面 THEN 系统 SHALL 展示与文章正文相关的图片素材，而不只展示封面素材或手工上传素材。
3. WHEN 素材来自正文图片上传流程 THEN 系统 SHALL 展示素材来源、关联文章、上传时间和当前状态。
4. IF 某个正文图片上传微信失败 THEN 系统 SHALL 在素材库中标记失败状态并提供可识别的失败原因。
5. WHEN 同一篇文章包含多张图片 THEN 系统 SHALL 为每张图片分别建立独立素材记录，而不是仅保留单一结果。
6. IF 正文图片已被微信托管并替换为新地址 THEN 系统 SHALL 保留原始来源地址与微信侧地址之间的映射关系以便排查问题。

### 需求 2

**用户故事：** 作为一名测试或开发人员，我希望本地 skill 可以直接对远端部署环境执行标准化测试，以便验证真实发布链路，而不是通过手工改数据库或临时改绑定关系来测试。

#### 验收标准
1. WHEN 需要验证远端部署环境的发布行为 THEN 系统 SHALL 支持通过本地 skill 指定远端服务地址和目标账号执行测试。
2. WHEN 本地 skill 发起测试请求 THEN 系统 SHALL 使用受控的接口参数表达目标公众号，而不是依赖修改 `agent.account_id`。
3. IF 远端环境返回错误 THEN 系统 SHALL 在测试结果中返回明确的失败阶段、错误信息和目标公众号标识。
4. WHEN 测试流程完成 THEN 系统 SHALL 保留请求参数与结果摘要，便于重复验证和问题复盘。
5. IF 测试请求仅用于验证草稿同步 THEN 系统 SHALL 支持在不影响其他公众号绑定关系的前提下完成测试。

### 需求 3

**用户故事：** 作为一名内容运营人员，我希望同一篇文章可以发送到多个公众号，以便复用内容并分别管理不同公众号的发布结果。

#### 验收标准
1. WHEN 用户对单篇文章选择多个目标公众号 THEN 系统 SHALL 为每个目标公众号分别执行独立的发送或草稿同步流程。
2. WHEN 单篇文章发送到多个公众号 THEN 系统 SHALL 不要求用户先修改 Agent 与公众号的绑定关系。
3. IF 某个目标公众号发送成功而另一个失败 THEN 系统 SHALL 分别记录各公众号的独立结果，而不是将整次操作视为单一成功或失败。
4. WHEN 用户查看文章发布结果 THEN 系统 SHALL 按公众号维度展示草稿 ID、发送状态、失败原因和最近更新时间。
5. IF 用户再次向同一公众号同步该文章 THEN 系统 SHALL 基于该公众号已有的草稿记录执行更新，而不是覆盖其他公众号的记录。
6. WHEN 用户首次为某篇文章向某个公众号发起同步 THEN 系统 SHALL 为该文章与该公众号建立独立的发布关系记录。

### 需求 4

**用户故事：** 作为一名系统管理员，我希望文章发布目标与 Agent 配置解耦，以便 Agent 负责内容生成，而公众号选择由发布动作决定。

#### 验收标准
1. WHEN 系统保存 Agent 配置 THEN 系统 SHALL 不将单一公众号绑定作为文章发布能力的唯一依据。
2. WHEN 用户执行文章发送或草稿同步 THEN 系统 SHALL 允许在请求层显式传入一个或多个目标公众号。
3. IF 某篇文章尚未建立任何公众号发布关系 THEN 系统 SHALL 允许用户直接选择目标公众号并创建新的发布关系。
4. WHEN 系统读取历史发布记录 THEN 系统 SHALL 以文章与公众号的关系记录作为查询基础，而不是仅依赖 Agent 当前绑定值。
5. IF Agent 默认配置了推荐公众号 THEN 系统 SHALL 将其仅作为默认选项，而不是强制唯一目标。

### 需求 5

**用户故事：** 作为一名内容运营人员，我希望多公众号发送过程中的媒体处理在每个公众号侧都可验证，以便确认正文图片、封面和草稿内容在目标公众号中一致可用。

#### 验收标准
1. WHEN 系统向目标公众号创建或更新草稿 THEN 系统 SHALL 针对该公众号完成封面与正文图片的媒体处理。
2. IF 同一篇文章被发送到多个公众号 THEN 系统 SHALL 不复用只对单一公众号有效的微信媒体标识。
3. WHEN 正文图片需要上传到微信 THEN 系统 SHALL 使用当前目标公众号对应的访问令牌执行上传。
4. IF 某个公众号的正文图片处理失败 THEN 系统 SHALL 阻止该公众号的草稿提交并记录失败详情，同时不影响其他公众号继续处理。
5. WHEN 用户查看某个公众号的文章结果详情 THEN 系统 SHALL 能看到该公众号对应的媒体处理结果摘要。

### 需求 6

**用户故事：** 作为一名产品负责人，我希望整个方案具备明确的成功标准和边界处理，以便在改造后稳定支持运营使用。

#### 验收标准
1. WHEN 用户对单篇文章选择多个公众号执行同步 THEN 系统 SHALL 在一次操作中返回每个公众号的独立处理结果。
2. IF 用户未选择任何目标公众号 THEN 系统 SHALL 阻止提交并提示必须至少选择一个公众号。
3. IF 用户选择了无权限或不可用的公众号 THEN 系统 SHALL 对该公众号返回明确错误且不修改其他公众号的结果。
4. WHEN 素材库中展示正文图片素材 THEN 系统 SHALL 支持按文章、公众号、素材状态进行检索或过滤。
5. WHEN 测试人员使用本地 skill 连接远端服务 THEN 系统 SHALL 提供不会破坏现有生产绑定关系的验证路径。
6. WHEN 本功能完成验收 THEN 系统 SHALL 满足“无需绑定解绑公众号、可看到正文相关素材、单文章可面向多个公众号独立发送”这三个核心成功标准。
